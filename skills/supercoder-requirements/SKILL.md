# Supercoder Requirements

The Requirements specialist gathers and documents what needs to be built.

## Responsibilities

### Requirements Gathering
- Interview stakeholders
- Analyze existing systems
- Research user needs
- Document assumptions

### Requirements Documentation
- Write user stories
- Define acceptance criteria
- Create feature specs
- Maintain requirements doc

### Requirements Analysis
- Identify scope
- Clarify ambiguities
- Prioritize features
- Identify dependencies

## Requirements Process

### 1. Discovery
1. Meet with stakeholders
2. Understand business goals
3. Identify user groups
4. List initial requirements

### 2. Analysis
1. Group related requirements
2. Prioritize by value/effort
3. Identify gaps
4. Resolve conflicts

### 3. Documentation
1. Write user stories
2. Define acceptance criteria
3. Create diagrams
4. Review with stakeholders

## User Story Template

```markdown
As a [user type],
I want to [feature],
So that [benefit].

## Acceptance Criteria
- [ ] Criteria 1
- [ ] Criteria 2
- [ ] Criteria 3
```

## Requirements Types

### Functional
- Features the system must have
- User interactions
- Data processing
- Business rules

### Non-Functional
- Performance requirements
- Security requirements
- Usability requirements
- Scalability requirements

## Best Practices

- **INVEST** - Independent, Negotiable, Valuable, Estimable, Small, Testable
- **Clear acceptance criteria** - Measurable, testable
- **Prioritize** - MoSCoW (Must, Should, Could, Won't)
- **Iterate** - Requirements evolve
